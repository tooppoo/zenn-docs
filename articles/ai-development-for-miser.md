# 吝嗇家とは何か

吝嗇家（りんしょくか）は、金や物を出し惜しむ人のことである。

つまるところ「ケチ」のことなのだが、「ケチ」という言葉を使うといささかネガティブな印象が強くなりすぎると感じたので、この言葉をわざわざ使っている。

一方で、単に「AI に金を払いたくない」という話にしたくなかった意図もある。コーディングエージェントは便利で、便利なものにはお金を払ってよい。
問題は、便利だからといって、まだ考えが固まっていない段階から雑に走らせると、判断の揺れまでエージェント利用料で支払うことになる点にある。

ここで扱うのは、単に「出費を抑える」という話でなく、「エージェント使用の効率を上げる」というテーマとも言える。

個人開発で月に数千円から数万円が飛ぶのは、十分に痛い。仕事で使う予算感と、個人で遊ぶ予算感は違う。
だからこそ、いかにして事前段階で問題を詰めて、どこからエージェントに渡すのかを分けたい。

# ChatGPTを使い倒す

結論から言えば、コーディングエージェントへタスクを渡す前に、ChatGPTを使って課題を磨き抜く、という方針になる。

なぜそこでChatGPTなのか。

自分の観測では、Claude は Claude Code と通常の Claude 利用が同じ消費枠で扱われる。
つまり、Claudeを使えばClaude Codeの使用量が増えるし、その逆も然りでらう。

一方で、ChatGPT は Codex と基本的には別枠[^1]となる。少なくとも、Codex 側の利用上限に到達した状態でも、ChatGPT から GitHub Issue を更新する作業はできていた。
この点（ChatGPT は Codex とは消費上限が別管理であること）に目をつけ、コーディングエージェントの前段としてChatGPTを活用しよう、というのが本稿の立場となる。

[^1]: ChatGPT 経由でコード修正をさせる場合や、ChatGPT for Excel、ChatGPT for PowerPoint のように裏側で Codex 系の処理が走るものは、Codex経由で実行され、しっかりCodexの使用量に計上されるらしいので、注意

# ChatGPTとレスバをする

最初にやるのは、ChatGPT への相談である。

やりたいこと、変えたいもの、直したいバグをとりあえず投げる。「こういうことをしたい」「こういう挙動が気に入らない」「この設計が怪しい気がする」くらいで始めてしまって良い。

ChatGPT からは、おそらくそれらしい案が返ってくるだろう。気になるところが特に無ければそれを踏まえてより具体化した方針を出すだろうが、どちらかというと、「イメージと合わない」「なんか違う」「よくわからない」を見つける方が重要だと考える。

「なんか違う」「イメージと合わない」ぐらいでも良いので、ズレや違和感、不明点を見つけ、それをChatGPTに打ち返し、ChatGPTがまた案を出す。こちらがまた引っかかる。
スライドではこれを雑に「レスバ」と呼んだ。

「レスバ」と言いつつ、その実体は、まだ言語化されていない前提を雑な案にぶつけて浮かび上がらせる作業である。

先述の通り、ChatGPT は Codex とは別管理であるようだから、ここでは遠慮なく迷って掌を返してよい。少なくともPlusプランなら、7~8タスクぐらい並行して議論し続けるでもしない限り、そうそう上限には引っかからない[^2]。

[^2]: とはいえ、そんな調子で数時間続けていると、たまに「リクエスト数が多すぎるので制限を入れた」というメッセージが出てきたりすることもある。

# Issue にして蓄積、またレスバ

やりたいことの輪郭がある程度見えてきたら、ChatGPT に Issue を立ててもらう。

ここで Issue 作られるIssueは、せいぜいチェックポイント程度の扱いで、また完成品ではない。
大抵の場合、そこからさらに吟味・検討の余地がある。吝嗇家ならば、ここでコーディングエージェントに投げるといささかトークン消費が勿体ない、と思いたい。

とりあえず Issue ができたら、自分で読む。気になるところがあれば、当然 ChatGPT に尋ねる。
気になるところが無くても、とりあえず雑に「Issue #38 をレビューして」と投げるぐらいでも、結構色々出てくる。

そうして、 ChatGPT が提案してきた内容や判断を求めてきた内容を確認して応答し、それをまたIssueに反映させる。
またIssueをレビューさせて、提案等に応答し、それをまたIssueに反映させる。これを「Issueの実装者が判断に迷わない」という評価が下るまで、ひたすら繰り返す。

自分の中では、Issue を Get Ready にする、という言い方をしている。

つまり、Issue が「エージェントに投げられる状態」になった状態、エージェントが迷う余地を減らし、迷ったときに参照できる判断材料が一通り Issue に置かれた状態を、 ChatGPT との「レスバ」を通じて目指す。

このくらいまで来れば、エージェントに渡してよい。逆に言えば、ここに達していない Issue は、まだ人間と ChatGPT で揉んだほうがよい。そこでケチる。

# エージェントにIssueを投げる

Issue が Get Ready になったら、ようやくコーディングエージェントを使う。この段階まで来れば、「Issue #38 を実装して」ぐらいの雑な指示でだいたいは十分になる。

個人では Claude Sonnet 5 と Codex 5.5 をメインで使用しているが、どちらのエージェントでも大体は最後まで完走してくれるし、内容もそれほど悪いものではない、というのが体感。

ここまできて初めて、金のかかる高い処理を走らせる。

もちろん、PR を読んで気になるところがあればコメントするし、コメント対応をまたエージェントに依頼する。
だが、最初から曖昧な依頼を投げて迷走させるよりは、ずっと安くつくと思う。

# Issue と PR に文脈を残す

この運用で重要なのは、会話ログに文脈を閉じ込めないことだ。

ChatGPT との会話は便利だが、チャットログにだけ方針が残っていると、あとで別のツールに引き継ぎにくい。
たとえば利用上限に当たって別のエージェントに乗り換えたくなったとき[^3]も、会話ログ依存だとつらい。

[^3]: 安いプランに入っている場合は、かなりの頻度で発生すると思う。少なくとも筆者は、1日1回は発生している

Issue と PR に文脈が残っていれば、「Issue #38 の作業を引き継いで」ぐらいで投げ直しやすい。

特別なオーケストレーション基盤やツールの類は要らないのが、この運用の手軽なところ。
GitHub Issue あるいは類似のIssue Tracking Systemさえあれば、同様のことができる。
人間、ChatGPT、コーディングエージェントのあいだでプロセス横断に共有できるメモリとして、Issueを利用する。

# AI時代のｎ ssue / Ticket Driven Development

これは、Issue Driven Development や Ticket Driven Development の素朴な再評価とも言えると思う。

人間同士なら雑な Issue でも割と会話で補えるが、エージェントに渡す場合、Issue に書かれていないことはかなりの確率で落ちるか、勝手に補われる。

勝手に補われたものが正しければよいが、正しくなかったとき、その修正にもまたコストがかかる。

だから Issue を書く。それも人間が読むためだけでなく、AI が読むことを前提に、対人なら嫌がられるほど執拗に修正を繰り返しながら書く。

とはいえ修正を繰り返してIssueの情報量が増えるほど良いというわけでもない。
求めているのは、判断が分かれる場所、失敗すると高くつく箇所、あとから揉めそうな箇所についての明示的な手がかりであり、かつ、それが共有可能な場所に置かれていることである。

# まとめ

- ChatGPT と Codex は消費トークンが別管理なのを利用して、往復の激しい部分はChatGPTを使い倒す
- AIエージェントが実装に迷わないレベルまで具体化・詳細化したら、最後の最後でエージェントに走らせる
- Issue / Ticket Drivenにすることで、複数ツールを連携する
- 途中で上限に引っかかったらエージェントを入れ替えて引き継がせる等も、特殊な仕組み無しで可能

コーディングエージェントは便利だが、高い。

だから、エージェントを使う局面を後ろに倒し、その前段で安上がりな方法を使う、というのが本稿の主旨である。

そして、その実践を支える仕組みとして、 Issue / Ticket Driven Development はこのAI時代において改めて評価されるべきかもしれない。
